# Company Rules

## 1. Communication 沟通制度

**规则：**
- 所有项目级指令统一由 nia 接收
- 所有执行级任务统一由 PM 下发
- 所有跨团队沟通默认通过 PM 或 Team Lead
- 所有升级问题必须写清：问题是什么、卡在哪里、需要谁决定
- 不允许无结论汇报，不允许"感觉差不多"式表达

**标准格式：**
```
Issue:
Impact:
Blocked By:
Suggested Fix:
Decision Needed:
```

---

## 2. Tasks 任务制度

**规则：**
- 所有任务必须结构化
- 任务必须有唯一 owner
- 任务必须写输入、输出、依赖、截止时间
- 没有验证方式的任务不能下发
- 无法执行的任务必须退回重写，不允许模糊硬做

**统一模板：**
```
TASK
Project:
Task:
Owner Agent:
Role:
Input:
Expected Output:
Dependencies:
Priority:
Deadline:
Validation Method:
```

**优先级：** P0=critical, P1=important, P2=enhancement

---

## 3. Document 文档制度

**规则：**
- 所有项目必须有 PRD
- 所有关键决策必须记录
- 所有上线项目必须有交付文档
- 所有项目结束必须有复盘
- 临时好用的角色必须沉淀进角色库
- 知识库不存闲聊流水账，只存结构化结果

---

## 4. Product 产品制度

**规则：**
- 所有 MVP 必须经过 Product Manager Agent 审查
- 核心功能默认不得超过 3 个
- 没有差异化，不进入研发
- 没有验证计划，不进入研发
- 没有清晰用户价值表达，不上线
- 产品上线前必须确认：值得发、值得用户试、值得被传播

---

## 5. SEO / Growth 增长制度

**规则：**
- SEO 是增长系统，不是附属工作
- 每个产品项目都要评估搜索流量机会
- 每个网站必须具备最基础 SEO 结构
- 每篇 SEO 内容必须有明确转化动作
- 不追求空流量，追求有业务价值的搜索流量

---

## 6. Engineering 研发制度

**规则：**
- 架构和开发分离
- 复杂问题必须单独建任务
- 不允许未评估就大改架构
- 优先选择成熟、稳定、低复杂度方案
- 先实现主路径，再补边界优化

---

## 7. Testing 测试制度

**规则：**
- 所有交付必须经过测试
- 关键流程必须有覆盖
- Bug 必须分级
- 没有测试结论，不得进入最终审核
- "能跑一次"不等于"可发布"

---

## 8. Security 安全制度

**规则：**
- 所有上线项必须经过安全审查
- 密钥、权限、依赖必须可追踪
- 严重风险未处理，不得上线
- 默认保守，默认最小权限
- 安全不是上线后再补的事

---

## 9. Release 发布制度

**规则：**
- Final Review Agent 是唯一发布批准者
- Deploy Agent 只执行，不审批
- 没有回滚方案，不得发布
- 没有文档，不得发布
- 没有 Product Manager 的"值得发"判断，不得发布

---

## 10. Retrospective 复盘制度

**规则：**
- 每个项目结束必须复盘
- 复盘至少回答：做对了什么、做错了什么、哪些可复用、哪些要禁止再犯
- 复盘必须进入共享记忆区
- 复盘后的规则应反哺制度或模板
