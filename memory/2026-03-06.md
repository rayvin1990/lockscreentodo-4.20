4. **确认流程** - 没有让 agent 复述需求就动手

### 经验教训

#### 问题 1：模型配置看官方文档
- 不要凭经验假设
- 查阅官方文档确认配置方法
- 避免配置错误导致功能异常

#### 问题 2：不要单个单个问题的改
- 要总结所有问题
- 一次性都输入给 agent
- 让 agent 复述一遍需求
- 反复确认没问题才执行
- **千万别着急动手**

#### 问题 3：描述不要有歧义
- 要精准具体
- 避免模糊词汇
- 用示例说明期望结果
- 明确指出不要什么

## 20:31 - 重要补充：沟通优先级

### 沟通层级（重要）

```
修改产品方向和文案逻辑：
1️⃣ 优先与大模型沟通（确定方向、策略、文案）
2️⃣ 其次与 Agent 沟通（执行具体任务）
```

### 为什么

| 类型 | 适合场景 |
|------|---------|
| **大模型** | 战略思考、文案创作、方向决策、产品定位 |
| **Agent** | 执行代码、文件操作、具体任务、重复工作 |

### 错误做法

❌ 直接让 Agent 改产品方向  
❌ 让 Agent 决定文案逻辑  
❌ Agent 边想边做，没有整体规划  

### 正确做法

✅ 先用大模型确定方向和文案  
✅ 再用 Agent 执行具体修改  
✅ Agent 执行前复述需求，确认无误

## 21:03 - 最重要补充：Commit 习惯

### 核心原则（最重要）

```
每次改完最终版 → 立即 Commit
```

### 为什么重要

- ✅ 有版本可回退
- ✅ 知道改了什么
- ✅ 避免改乱后无法恢复
- ✅ 方便对比差异

### Commit 时机

| 时机 | 操作 |
|------|------|
| 初始版本完成 | `git commit -m "Initial version"` |
| 每次修改完成 | `git commit -m "描述改动"` |
| 修复 Bug 后 | `git commit -m "Fix: 描述问题"` |
| 功能迭代后 | `git commit -m "Feature: 描述功能"` |

### 今天的教训

❌ 没有 Commit → 改乱了无法回退  
✅ 下次：每次改完立即 Commit

### Commit 规范

```bash

git add .
```

## 21:11 - 最重要原则：避免完美主义

### 核心原则（说三遍）

```
避免完美主义
避免完美主义
避免完美主义
```

### 核心理念

```
能用就好 ✅
够用就行 ✅
迭代优化 ✅
```

### 今天的教训

❌ 悬停功能已经够用了 → 还要加自动显示  
❌ 自动显示够用了 → 还要加划线检测  
❌ 划线检测够用了 → 还要加动态加载  
❌ 越改越复杂 → 最后回到最初版本  

### 正确做法

```
1. 确定 MVP（最小可用功能）
2. 实现 MVP
3. 测试可用
4. Commit 保存
5. 有需求再迭代
```

### 判断标准

| 问题 | 答案 |
|------|------|
| 能用吗？ | 能 → OK |
| 有 Bug 吗？ | 没有 → OK |
| 用户满意吗？ | 满意 → OK |
| 需要完美吗？ | 不需要 → 发布 |

### 悬停 vs 自动显示

**今天原本：** 悬停显示 ✅（够用）  
**后来改成：** 自动显示 ❌（完美主义）  
**结果：** 越改问题越多  

**教训：** 悬停就够了，别动
